# humanize

Rewrite wiki prose to read like a thoughtful person wrote it. The standalone counterpart to the write-time discipline in `skills/wiki/references/writing-style.md`.

## Reading list (every invocation)

1. `skills/wiki/references/writing-style.md` — the spec for what to fix and what to leave alone
2. `wiki/meta/voice-sample.md` — IF it exists, use it as a style reference (sentence rhythm, vocabulary preferences, structural habits)
3. The target page(s)

## What humanize PRESERVES (never modifies)

- The entire frontmatter block (`type`, `title`, `tags`, `summary`, `provenance`, `confidence`, type-specific fields)
- All `[[wikilinks]]` and their text
- All inline provenance markers: `^[inferred]`, `^[ambiguous]`
- All custom callouts and their content type: `[!contradiction]`, `[!gap]`, `[!key-insight]`, `[!stale]`, `[!ai-writing]`
- All code blocks (```...```), inline code (`...`), and their contents
- All wiki page citations and source links
- The factual content — only the prose around it gets rewritten

## What humanize CHANGES

- AI vocabulary words → plainer alternatives (per writing-style.md table)
- Copula avoidance → plain `is`/`are`
- "Not just X, but Y" → direct statement
- Em-dash overuse → mix of commas, parens, colons
- Inline-header bolded lists (`**Term:** description`) → proper definition list, prose, or shorter bullets
- Title Case headings → sentence case
- Significance inflation → factual statement
- Vague attributions → specific cite or deletion
- Filler phrases → deletion
- Excessive hedging → direct statement
- Generic wrap-up conclusions → deletion (page is done when it's done)
- Sycophantic openers → deletion

## What humanize SKIPS (never touches)

- Pages with `type: source` — extracted content, not synthesized
- Pages where `provenance.extracted >= 0.9` — mostly direct extraction
- Pages tagged `humanize/skip` (manual override)
- The contents of `.raw/` (immutable in any case)
- Auto-generated sections inside `<!-- AUTO-... -->` markers (those get regenerated by their owning skill)

## Modes

### Mode 1: single-page (`humanize <page>`)

Rewrite one page in place.

#### Workflow

1. Read writing-style.md and (if exists) voice-sample.md.
2. Read the target page.
3. If page is a `source` type or `provenance.extracted >= 0.9`, abort with: "Skipping — extracted content shouldn't be rewritten. If you want to rewrite anyway, pass `--force`."
4. Section-by-section, scan for the patterns in writing-style.md.
5. Rewrite each affected section. Keep the meaning identical.
6. Run a self-audit pass — re-scan for AI tells in the new prose.
7. Show the user a diff summary: `12 changes (3 vocabulary, 4 copula, 2 em-dash, 1 filler, 2 inline-bold lists)`.
8. Save. Bump `updated:` in frontmatter.
9. Recompute `provenance.inferred` if rewrites were substantive (humanize itself adds inference). Add `^[inferred]` markers if your rewrite restated something the original source didn't say verbatim.
10. Run `cross-linker` scoped to the page (some rewrites may have moved or removed wikilink-eligible text).
11. Append to `wiki/log.md`.

### Mode 2: batch by pattern (`humanize batch --pattern <pattern> [--scope <scope>]`)

Fix one named pattern across many pages.

Patterns supported:
- `vocabulary` — AI vocabulary words
- `copula` — copula avoidance
- `em-dash` — em-dash overuse (above 1/paragraph density)
- `negative-parallel` — "not just X, but Y"
- `inline-bold-list` — `**Term:** description` lists
- `title-case-headings` — Title Case in body
- `filler` — filler phrases ("in order to", etc.)
- `hedging` — excessive hedging
- `wrap-up-conclusions` — generic "In conclusion" paragraphs
- `sycophantic` — sycophantic openers
- `significance-inflation` — manufactured importance
- `vague-attributions` — "researchers", "experts" etc.

Scope:
- Default: all pages with `provenance.inferred > 0.4` (synthesis-heavy)
- `--scope all` — every page in `wiki/` (still skips source-type and high-extraction pages)
- `--scope <folder>` — restrict to a folder
- `--scope tag:<tag>` — restrict to a tag
- `--scope flagged-by-lint` — only pages with `[!ai-writing]` callouts from a recent lint run

#### Workflow

1. Read writing-style.md and voice-sample.md (if exists).
2. Build the in-scope page list per the scope filter.
3. For each page, scan for the named pattern only. Skip pages where the pattern doesn't appear.
4. Rewrite the affected sentences/paragraphs. Don't fix unrelated patterns in this run (one pass = one pattern, easier to review).
5. Show progress per page; summary at end (`X pages touched, Y rewrites total`).
6. Bump `updated:` on each touched page.
7. Run `cross-linker` once across all touched pages.
8. Append a single batch entry to `wiki/log.md`:

```
## [YYYY-MM-DD HH:MM] humanize | batch --pattern <p> --scope <s>
- pages_scanned: N
- pages_touched: M
- rewrites: K
- voice_calibration: yes|no (sample present)
```

### Mode 3: voice calibration (`humanize calibrate [--auto] [--from <source>]`)

Helper to set up voice calibration.

#### Default (interactive)

1. Check if `wiki/meta/voice-sample.md` exists. If yes, show its current contents and ask if user wants to replace.
2. Prompt the user to paste 2-3 paragraphs of their own writing.
3. Save to `wiki/meta/voice-sample.md` with universal frontmatter (`type: meta`, `summary: "User voice sample for humanize calibration"`, `visibility/internal` — never export).
4. Confirm: "Voice sample saved. Future humanize runs will match this style."

#### `--auto` (sample existing user content from the vault)

When the user has already written enough in the vault, you don't need to prompt — sample existing pages.

1. Build a candidate pool from the vault. Eligibility:
   - `type` in `{journal, meeting, decision, question, project}` (user-authored types)
   - `provenance.extracted >= 0.9` (mostly the user's own writing, not synthesis)
   - Body length ≥ 500 chars (enough material to sample)
   - NOT in `wiki/_raw/` (drafts haven't been polished)
   - NOT tagged `humanize/skip`
2. If pool has < 3 pages → fall back to interactive mode and tell the user why ("not enough user-authored content yet — please paste a sample").
3. Otherwise, sample 2-3 paragraphs from the pool:
   - Prefer the most recent pages (last 60 days)
   - Across at least 2 different page types if possible
   - Skip frontmatter, code blocks, and quoted material
   - Concatenate into ~500-1000 words total
4. Save to `wiki/meta/voice-sample.md` with `type: meta`, tag `visibility/internal`, summary: `"Auto-sampled from N user-authored pages, ISO-date"`.
5. Add a frontmatter field `sampled_from: [list of pages used]` so the user can re-run if they update those pages.
6. Confirm: "Voice sample auto-built from N pages: [list]. Re-run with `--auto` anytime to refresh."

#### `--from <source>` (sample from a specific page or folder)

Take an explicit source to sample from instead of the default pool.

- `humanize calibrate --from "wiki/journal"` — sample from all journal entries
- `humanize calibrate --from "wiki/journal/2026-04-15-deep-work.md"` — single page
- `humanize calibrate --from "wiki/projects/myproject"` — sample a project's contents

Same workflow as `--auto` but scoped to the supplied path.

#### Refresh

The sample stale-detects on use: if `wiki/meta/voice-sample.md`'s `sampled_from:` pages have been updated since the sample was built, humanize will warn ("voice sample is stale — re-run `humanize calibrate --auto`?"). User can ignore or refresh.

## Voice calibration: what it does

When `wiki/meta/voice-sample.md` exists, humanize reads it BEFORE rewriting and biases the output toward:
- Average sentence length found in the sample
- Vocabulary register (formal vs. conversational vs. technical)
- Structural habits (how often sample uses lists, em dashes, parens, etc.)
- Personal tics that you actually like (not all repetition is bad)

Without a voice sample, humanize defaults to "plain, direct, factual" — appropriate for most wiki content but neutral.

The sample stays in the vault (you can edit it), is tagged `visibility/internal` so it's never exported, and only humanize reads it.

---

## When NOT to humanize

- Pages someone else wrote (sources, transcripts, paper summaries) — those should stay in the original voice
- Pages that already follow writing-style.md — re-running adds noise
- Pages tagged `humanize/skip`
- Auto-generated sections (`<!-- AUTO-GENERATED -->` markers)
- The current session's hot.md (wait for SessionStop hook to refresh it first)

## Integration with other skills

- **wiki-ingest**: writing-style.md is consulted at write-time. Heavy synthesis pages get marked for humanize.
- **wiki-lint Check 13**: flags pages with AI tells. The lint report names them; humanize fixes them.
- **save**: same write-time discipline.
- **autoresearch**: synthesis pages skew inferred — heaviest candidates for periodic humanize batches.
- **wiki-rebuild**: after a full rebuild, run `humanize batch --scope all` if the previous content was AI-tell-heavy.
- **cross-linker**: runs after humanize because rewrites can move or remove wikilink-eligible text.

---

## A note on respect for the original

When a page mixes extracted and inferred content, ONLY rewrite the inferred parts. Preserve direct quotes verbatim (they should already be in `>` blockquotes). When in doubt, leave it alone and flag for review instead of rewriting.

The risk of humanize is it sounds confident and rewrites something the original source said precisely. The mitigation is the source pages (`type: source`) are always excluded, and the universal `provenance:` block lets readers know which pages are synthesis-heavy and therefore most "rewritten."
